# LocalWiki — Product Requirements Document
### Karpathy-Style Self-Compiling Knowledge Management System
**Version:** 1.1 | **Target Implementer:** Claude Code | **Date:** 2026-05-01

***

## 1. Executive Summary

LocalWiki is a fully local, Python-based web application implementing Andrej Karpathy's "LLM Knowledge Bases" pattern. It replaces RAG-style search with a persistent, self-compiling wiki of interlinked Markdown files maintained by a local LLM (Ollama `gemma4:e4b`). Users can upload documents via drag-and-drop, navigate the compiled wiki, chat with their knowledge base, and optionally challenge or enrich it with live web research through a ReAct agentic system.[1]

**Core principle:** *Keep It Simple*. When in doubt, the simplest working solution wins. No vector database, no LangChain, no external APIs except Tavily for the optional web research feature.[1]

**Update for this version:** The implementation must not force Streamlit as the UI framework; it must use `uv` for Python environment and dependency management; the GUI must follow a New York Times-inspired visual style; and the automatic test suite must be capped at 100 tests, with approximately 90% covering the most important system behavior and the remainder covering newly implemented features.[1]

***

## 2. System Architecture Overview

### 2.1 Three-Layer Knowledge Model (Karpathy Pattern)

```
data/
├── raw/                    # Layer 1: Immutable source documents (LLM reads, never writes)
│   ├── uploads/            # User-uploaded files (original format)
│   ├── extracted/          # Plain-text versions extracted from PDF/DOCX/etc.
│   └── .manifest.json      # Deduplication registry (SHA-256 hashes)
└── wiki/                   # Layer 2: LLM-generated markdown (LLM owns entirely)
    ├── index.md            # Master catalog — updated on every ingest
    ├── log.md              # Append-only activity log
    ├── overview.md         # High-level synthesis of all knowledge
    ├── concepts/           # Concept/topic pages
    ├── entities/           # People, organizations, projects
    ├── sources/            # Per-document summary pages
    └── comparisons/        # Cross-document analysis pages (filed from queries)
```

**Layer 3 — Schema** is stored in `SCHEMA.md` at project root. It is injected into every LLM system prompt so the model behaves as a disciplined wiki maintainer, not a generic chatbot.[1]

### 2.2 Application File Structure

```
localwiki/
├── app.py                  # Web application entry point (main UI)
├── wiki_engine.py          # Core: ingest(), query(), lint() operations
├── ollama_client.py        # Thin wrapper around ollama Python SDK
├── agent.py                # ReAct web research agent loop
├── tools.py                # Tool definitions: tavily_search, report_writer
├── file_processor.py       # Extracts plain text from PDF, DOCX, MD, TXT, HTML
├── dedup.py                # SHA-256 hash-based deduplication logic
├── schema_loader.py        # Loads and returns SCHEMA.md content
├── SCHEMA.md               # Wiki schema injected into all LLM system prompts
├── pyproject.toml          # Project metadata and dependencies managed with uv
├── uv.lock                 # Locked dependency set generated by uv
├── .env.example            # Template: TAVILY_API_KEY=...
├── tests/                  # Automated test suite (max 100 tests)
└── data/                   # All knowledge data (gitignore raw/ optionally)
    ├── raw/
    └── wiki/
```

### 2.4 GUI Design Direction

The graphical user interface shall follow a **New York Times-style** visual language. This means the UI must feel editorial, calm, highly readable, and content-first rather than dashboard-heavy or overly technical.[1]

**Design requirements:**
- Typography must prioritize readability and hierarchy, using a serif-forward or editorial-inspired combination for key headings and a clean highly legible body font.
- Layout must use generous whitespace, restrained color usage, and strong typographic structure.
- Visual emphasis must come from spacing, line rules, scale, and hierarchy rather than bright accent colors or dense card grids.
- Surfaces should remain minimal and quiet; avoid playful SaaS aesthetics, neon accents, or overly rounded component styles.
- The interface must present documents, wiki pages, citations, and research outputs as if they are being read in a premium digital publication.
- Any chosen frontend framework or component system must be adapted to this style rather than imposing its default visual identity.

## 3. Functional Requirements

### 3.1 Module: `dedup.py` — Deduplication Engine

**Purpose:** Prevent duplicate documents from being ingested into the knowledge base.[1]

**Data structure:** `data/raw/.manifest.json`
```json
{
  "a3f4c8...": {
    "filename": "my_paper.pdf",
    "original_name": "my_paper.pdf",
    "added_at": "2026-04-29T21:00:00",
    "extracted_path": "data/raw/extracted/my_paper.txt",
    "size_bytes": 48200
  }
}
```

**Functions:**

```python
def compute_hash(file_bytes: bytes) -> str:
    """Returns SHA-256 hex digest of file bytes."""

def load_manifest() -> dict:
    """Loads .manifest.json; returns empty dict if not found."""

def save_manifest(manifest: dict) -> None:
    """Saves manifest to .manifest.json atomically."""

def is_duplicate(file_bytes: bytes) -> tuple[bool, str | None]:
    """
    Returns (True, existing_filename) if duplicate found,
    (False, None) otherwise.
    """

def register_file(file_bytes: bytes, filename: str, extracted_path: str) -> str:
    """
    Adds file to manifest. Returns hash string.
    Must be called AFTER is_duplicate check returns False.
    """
```

**Rules:**
- Hash computed over raw file bytes (not filename)
- Manifest must be updated atomically (write to temp file, rename)
- Duplicate detection is hash-exact, no fuzzy matching

### 3.2 Module: `file_processor.py` — File Text Extraction

**Purpose:** Convert uploaded files of any supported type into clean plain text for LLM consumption.[1]

**Supported formats:**
- `.txt`, `.md` → Read as-is (UTF-8)
- `.pdf` → Extract with `pypdf.PdfReader`; join pages with `\n\n---\n\n`
- `.docx` → Extract paragraphs with `python-docx`
- `.html` → Strip tags with `re` (no external deps); keep text nodes
- Other formats → Raise `UnsupportedFileTypeError`

**Functions:**

```python
def extract_text(file_path: Path) -> str:
    """
    Returns extracted plain text from file.
    Raises UnsupportedFileTypeError for unsupported types.
    Max output: 50,000 characters (truncate with notice if exceeded).
    """

def save_extracted(text: str, original_filename: str) -> Path:
    """
    Saves extracted text to data/raw/extracted/<stem>.txt
    Returns path to saved file.
    """
```

**Rules:**
- Never modify files in `data/raw/uploads/` — they are immutable sources
- If extraction fails partially (e.g., encrypted PDF), save what was extracted and log a warning
- Truncation threshold: 50,000 characters with a `[TRUNCATED at 50000 chars]` marker appended

**Purpose:** Single point of contact for all LLM calls. Wraps the `ollama` Python SDK.[1]

**Model:** `gemma4:e4b` (hardcoded as default, configurable via `OLLAMA_MODEL` env var)[1]

**Functions:**

```python

### 3.5 File: `SCHEMA.md` — Wiki Schema

This file is the "brain" of the system. It must be created at project setup. Content remains unchanged unless implementation details require path updates only.[1]

```markdown
# LocalWiki Schema

You are a disciplined wiki maintainer. Your job is to maintain a structured,
interlinked knowledge base in Markdown format. You follow these rules exactly.

## Directory Structure
- `data/raw/extracted/` — Plain text source documents. READ ONLY. NEVER MODIFY.
- `data/wiki/` — Your wiki. You own this entirely. Create and update pages here.
- `data/wiki/index.md` — Master catalog. UPDATE ON EVERY INGEST.
- `data/wiki/log.md` — Append-only activity log. APPEND ONLY, NEVER EDIT.

## Page Conventions
Every wiki page MUST start with YAML frontmatter:
```yaml
***
title: "Page Title"
type: concept | entity | source-summary | comparison
sources: ["raw/extracted/file1.txt"]
related: ["wiki/concepts/other-page.md"]
created: "YYYY-MM-DD"
updated: "YYYY-MM-DD"
confidence: high | medium | low
***
```

Use [[wikilinks]] for cross-references between pages (e.g., [[concepts/attention]]).

## index.md Format
Organize by category. Each entry: `- [[path/to/page]] — one-line description (N sources)`
Example:
```
## Concepts
- [[concepts/transformer-architecture]] — Self-attention and feed-forward blocks (3 sources)
## Entities
- [[entities/openai]] — AI lab, GPT series (5 sources)
## Source Summaries
- [[sources/paper-2026-01]] — Summary of attention survey paper
```

## log.md Format
Each entry: `## [YYYY-MM-DD HH:MM] TYPE | Title`
Types: ingest | query | lint | research
Example: `## [2026-04-29 21:00] ingest | My Research Paper`

## Ingest Workflow
When asked to ingest a document:
1. Read the extracted text from data/raw/extracted/
2. Create a source summary page in data/wiki/sources/
3. Create or UPDATE (do not duplicate) relevant concept pages in data/wiki/concepts/
4. Create or UPDATE relevant entity pages in data/wiki/entities/
5. If the document contradicts existing pages, FLAG the contradiction clearly
6. Update data/wiki/index.md with any new pages
7. Append an entry to data/wiki/log.md
8. Return a JSON summary: {"pages_created": [...], "pages_updated": [...], "contradictions": [...]} 

## Query Workflow
When asked a question:
1. Read data/wiki/index.md to identify relevant pages
2. Read those pages
3. Synthesize a direct answer with [[wikilink]] citations
4. End response with: "Want me to file this as a wiki page? (yes/no)"

## Lint Workflow
When asked to lint:
Scan all wiki pages and return a markdown report with sections:
- ## Contradictions (pages with conflicting claims)
- ## Orphan Pages (no inbound links)
- ## Missing Pages (concepts mentioned but no page exists)
- ## Stale Claims (claims that may be outdated)
- ## Suggested Investigations (gaps in knowledge)
```

### 3.6 Module: `wiki_engine.py` — Core Wiki Operations

**Purpose:** Implements the three core Karpathy operations: Ingest, Query, Lint.[1]

#### 3.6.1 `ingest(extracted_text_path: Path, original_filename: str) -> dict`

```
Flow:
1. Read extracted text from extracted_text_path
2. Read current data/wiki/index.md (if exists)
3. Build messages list:
   - system: get_ingest_system_prompt() (includes full SCHEMA.md)
   - user: f"Please ingest the following document named '{original_filename}':\n\n{text}\n\nCurrent wiki index:\n\n{index_content}"
4. Call ollama_client.chat_text(messages, temperature=0.3)
5. Parse the LLM response:
   - Extract wiki page contents from response (delimited by ```markdown ... ``` blocks or clear headers)
   - Write each page to the correct wiki subdirectory
   - Update data/wiki/index.md
   - Append to data/wiki/log.md
6. Return summary dict: {"pages_created": [...], "pages_updated": [...], "contradictions": [...]} 
```

**Rules:**
- If `index.md` exceeds 8,000 characters, truncate to last 8,000 chars with a `[...truncated...]` marker
- Page writing: parse LLM output looking for `### FILE: wiki/concepts/xxx.md` headers followed by content blocks; write each file; fall back to writing a single source summary if parsing fails
- Log entry format must match SCHEMA.md exactly
- Return dict even on partial failure (include error message in `errors` key)

**LLM Output Format Prompt Addition:**
Append to ingest user message:
```
Respond ONLY in this format. For each file to create/update, use:
### FILE: data/wiki/sources/summary-<slugified-filename>.md
<full markdown content with YAML frontmatter>

### FILE: data/wiki/concepts/<concept-name>.md
<full markdown content>

...

### INDEX_UPDATE
<complete new content of data/wiki/index.md>

### LOG_ENTRY
## [<ISO datetime>] ingest | <document title>
Source: data/raw/extracted/<filename>
Pages created: <list>
Pages updated: <list>
```

#### 3.6.2 `query(question: str, file_as_page: bool = False) -> str`

```
Flow:
1. Read data/wiki/index.md
2. Build messages:
   - system: get_query_system_prompt()
   - user: f"Index:\n\n{index_content}\n\nQuestion: {question}"
3. Call ollama_client.chat_text(messages, temperature=0.7)
4. If file_as_page is True:
   - Extract answer as new comparison/analysis page
   - Write to data/wiki/comparisons/<slugified-question>.md
   - Update index.md and log.md
5. Return answer string
```

**Relevant page loading:** After step 1, parse `index.md` to extract wiki page paths mentioned. Read up to 5 most relevant wiki pages (heuristic: pages whose title appears in the question). Append their content to the user message. Keep total context under ~12,000 characters.[1]

#### 3.6.3 `lint() -> str`

```
Flow:
1. Collect all wiki page paths from data/wiki/ recursively
2. Read index.md and log.md
3. Read all wiki pages (concatenate with filename headers)
4. Build messages:
   - system: get_lint_system_prompt()
   - user: f"Please lint the wiki. All pages:\n\n{all_pages_content}"
5. Return LLM response (markdown health report)
6. Append lint entry to log.md
```

**Rules:** If total wiki content exceeds 15,000 characters, send only index.md + a summary of page count/types, not all page contents.[1]

### 3.8 Module: `agent.py` — ReAct Web Research Agent

**Purpose:** Implements a ReAct (Reason + Act) loop that uses `tavily_search` and `report_writer` as tools. Pattern mirrors the intended deep-agents-style workflow in the original PRD.[1]

**Architecture:** Manual ReAct loop — no LangChain, no framework. Uses Ollama native tool calling (`tool_calls` in response).[1]

```
ReAct Loop:
1. Initialize messages with system prompt + user research question
2. Call ollama_client.chat() with tools=TOOLS
3. If response has tool_calls:
   a. For each tool_call:
      - Extract function name and arguments
      - Execute the Python function (tavily_search or report_writer)
      - Append tool result to messages as role="tool"
   b. Yield intermediate step to the UI (for live display)
   c. If report_writer was called → loop ends (final report written)
   d. Else if max_iterations reached → break with partial result
   e. Goto step 2
4. If response has no tool_calls → final answer, return it
5. Max iterations: 8 (safety limit)
```

#### 3.8.1 `run_research_agent(question: str, wiki_context: str = "") -> Generator`

```python
def run_research_agent(
    question: str,
    wiki_context: str = ""
) -> Generator[dict, None, None]:
    """
    Runs the ReAct research agent.
    Yields step dicts for live UI display:
    {"type": "thought", "content": "..."}
    {"type": "tool_call", "name": "tavily_search", "args": {...}}
    {"type": "tool_result", "name": "tavily_search", "result": "..."}
    {"type": "final_answer", "content": "...", "report_path": "..."}
    {"type": "error", "content": "..."}
    """
```

**System prompt for agent:**
```
You are a research agent with access to web search. Your goal is to research
the given question thoroughly and produce a structured report.

{{wiki_context_if_provided}}

You have two tools:
- tavily_search: Search the web for information
- report_writer: Save your completed research report to the knowledge wiki

Workflow:
1. Break the question into sub-questions
2. Search for each sub-question (use tavily_search multiple times)
3. Reflect on results: do you have enough information?
4. If not, search again with refined queries
5. When you have sufficient information, call report_writer with a complete,
   well-structured markdown report including all sources
6. After report_writer is called, your work is done

Always cite your sources in the report using [Source: URL] format.
Max searches: 6. Be efficient.
```

#### 3.8.2 Reflection mechanism

After each set of tool results, before the next LLM call, append a reflection prompt:
```
Based on the search results above, reflect:
- Do you have enough information to write the report?
- What gaps remain?
- What should you search for next?
Proceed with your next action.
```

## 4. Non-Functional Requirements

### 4.1 Startup & Initialization

On first run, the application must:[1]
1. Create `data/raw/uploads/`, `data/raw/extracted/`, `data/wiki/concepts/`, `data/wiki/entities/`, `data/wiki/sources/`, `data/wiki/comparisons/` if they don't exist.
2. Create empty `data/wiki/index.md`, `data/wiki/log.md`, `data/wiki/overview.md` if they don't exist.
3. Check Ollama connectivity via `ollama_client.is_available()` and show visible offline status if unavailable.
4. Copy `SCHEMA.md` if it doesn't exist (from embedded default).
5. Verify that the project environment is being run through `uv`-managed dependencies and that setup instructions assume `uv` from the start.

### 4.2 Error Handling

| Error | Handling |
|---|---|
| Ollama not running | Show visible offline indicator; disable ingest/chat/lint actions; show instruction: `ollama serve` |
| `gemma4:e4b` model not found | Show error: `ollama pull gemma4:e4b` |
| TAVILY_API_KEY missing | Disable or gate the Research page; show setup instructions |
| PDF extraction fails | Save partial text, log warning, show user: `Partial extraction: N pages read` |
| LLM response parsing fails | Save raw LLM response as `data/wiki/sources/raw-ingest-<timestamp>.md`; do not crash |
| Wiki page write fails | Show error, do not corrupt existing pages |
| Dependency environment mismatch | Show setup guidance instructing the user to run the project with `uv sync` and the documented `uv run ...` commands |

### 4.3 Performance

- Ingest response time: expected 30-120s depending on document size and hardware.[1]
- Chat response time: expected 10-30s.[1]
- No artificial request timeouts required beyond framework defaults; let the chosen web stack and Ollama handle their own limits unless implementation evidence suggests otherwise.[1]
- Wiki search (full-text): must complete in < 1s for wikis up to 500 pages.[1]

### 4.4 Simplicity Rules

- **One Python file per module** — no sub-packages.[1]
- **No async code** — synchronous everywhere unless the chosen UI stack strictly requires async boundaries, in which case the public product behavior must still remain simple.[1]
- **No database** — files and JSON only.[1]
- **No Docker** — local setup only unless explicitly added later.[1]
- **No configuration UI** — `.env` file for all config.[1]
- **Use `uv` for environment and dependency management** — no `pip install -r requirements.txt` setup path in the primary workflow.
- **Do not force a frontend framework** — choose the simplest implementation that satisfies the UX and style requirements.

### 4.5 Testing Strategy

The project must include automated tests, but the total number of automatic tests must be capped at **100 tests maximum**. This cap is intended to keep the suite intentionally lean, fast, and focused.[1]

**Test allocation rules:**
- Approximately **90%** of tests shall cover the most important parts of the system.
- The remaining approximately **10%** shall cover newly implemented features.
- Priority coverage areas include deduplication, file extraction, Ollama connectivity behavior, wiki ingest/query/lint core flows, and critical error handling.
- Avoid broad low-value test proliferation, snapshot sprawl, or exhaustive UI micro-tests.
- Prefer a compact suite of high-signal unit and integration tests over a large volume of shallow tests.

**Suggested distribution at the cap:**
- Up to 90 tests for critical and high-risk core behavior.
- Up to 10 tests for newly added or recently changed features.

## 7. Detailed Data Flow Diagrams

### 7.1 Ingest Flow

```
User drops document
     │
     ▼
[dedup.py] is_duplicate(bytes)?
     │
     ├─ YES → Show warning "Already ingested on <date>" → STOP
     │
     └─ NO ──► [dedup.py] register_file()
                    │
                    ▼
              [file_processor.py] extract_text(path) → plain text
                    │
                    ▼
              [wiki_engine.py] ingest()
                    │
                    ▼
              Build prompt: SCHEMA + index.md + extracted text
                    │
                    ▼
              [ollama_client.py] chat(messages, temperature=0.3)
              (calls gemma4:e4b locally via Ollama)
                    │
                    ▼
              Parse LLM response: extract FILE: blocks
                    │
                    ▼
              Write wiki pages to data/wiki/**/*.md
                    │
                    ▼
              Update index.md + append to log.md
                    │
                    ▼
              Return {pages_created, pages_updated, contradictions}
                    │
                    ▼
              UI shows ingest summary
```

### 7.2 ReAct Agent Flow

```
User enters research question
          │
          ▼
[agent.py] run_research_agent(question, wiki_context)
          │
          ▼
     messages = [system_prompt, {role: user, content: question}]
          │
    ┌─────▼──────────────────────────────────────────────┐
    │         REACT LOOP (max 8 iterations)             │
    │                                                   │
    │  [ollama_client.py] chat(messages, tools=TOOLS)   │
    │              │                                    │
    │              ▼                                    │
    │  response has tool_calls?                         │
    │       │                      │                    │
    │      YES                     NO                   │
    │       │                      │                    │
    │       ▼                      ▼                    │
    │  For each tool_call:    Return content as         │
    │  - if tavily_search:    final_answer              │
    │    call tools.py fn     (exit loop)               │
    │    → yield tool_call step                         │
    │    → yield tool_result step                       │
    │    → append to messages                           │
    │  - if report_writer:                             │
    │    save report to wiki                           │
    │    → yield final_answer step                     │
    │    → EXIT LOOP                                   │
    │                                                   │
    │  Append reflection prompt to messages            │
    │  → continue loop                                 │
    └───────────────────────────────────────────────────┘
          │
          ▼
    UI renders each step live
```

## 10. Key Design Decisions & Rationale

| Decision | Rationale |
|---|---|
| `index.md` instead of vector DB | Per Karpathy-style pattern, this works well at small-to-medium source counts with zero infrastructure overhead.[1] |
| `gemma4:e4b` via Ollama | Native tool calling support and local-first execution fit the product goals.[1] |
| Manual ReAct loop, no LangChain | Simpler, fewer dependencies, full transparency, easier debugging.[1] |
| SHA-256 deduplication | O(1) lookup, deterministic, no false positives.[1] |
| UI framework not forced | Keeps implementation flexible and avoids constraining the developer to Streamlit when another simple web stack is a better fit.[1] |
| New York Times-style GUI | Matches the product's document-centric, reading-heavy workflow and supports a high-trust editorial presentation.[1] |
| `uv` for environment management | Produces a reproducible, modern Python workflow with locked dependencies and simple command execution.[1] |
| Max 100 automated tests | Keeps the suite fast and disciplined while forcing attention onto the most important system behavior and newly added features.[1] |
| SCHEMA.md as system prompt | Makes LLM behavior reproducible across sessions.[1] |
| Files over database | Every wiki page remains human-readable, git-trackable, and low-admin.[1] |
